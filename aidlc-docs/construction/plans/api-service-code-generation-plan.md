### Step 2: Data Models Generation
- [ ] Generate SQLModel data models (BlogPost, Tag, relationships)
- [ ] Generate Pydantic request/response models
- [ ] Ensure field validation constraints (max lengths, required fields)
- [ ] Implement many-to-many relationship between BlogPost and Tag

**Story Coverage**: All CRUD operations

**File**: `app/models.py`

**Models to generate**:
1. BlogPost (SQLModel, table=True)
   - id: auto-generated primary key
   - title: required string, max 500 chars
   - content: required string, max 10000 chars
   - tags: relationship to Tag (many-to-many)
   - created_at: auto-generated timestamp
   - updated_at: auto-generated timestamp

2. Tag (SQLModel, table=True)
   - id: auto-generated primary key
   - name: required string, max 50 chars, unique
   - posts: relationship to BlogPost (many-to-many)

3. BlogPostCreate (Pydantic)
   - title: required, max 500 chars
   - content: required, max 10000 chars
   - tags: optional list of strings

4. BlogPostUpdate (Pydantic)
   - title: optional, max 500 chars
   - content: optional, max 10000 chars
   - tags: optional list of strings

5. BlogPostRead (SQLModel)
   - All BlogPost fields for responses

6. PaginationResponse (Pydantic)
   - items: list of BlogPost
   - next_cursor: int or None
   - has_more: bool

### Step 10: API Endpoints Generation - GET /posts (List)

- [ ] Return PaginationResponse with next_cursor and has_more

**Story Coverage**: List blog posts with pagination and tag filtering

**Implementation**:
- Route: GET /posts
- Query parameters:
  - cursor: optional int (default None)
  - limit: optional int (default 20, max 100)
  - tag: optional string (default None)
- Output: PaginationResponse
- Status: 200 OK
- Pagination logic:
  1. Load all posts with tags
  2. Filter by tag if provided
  3. Sort by ID
  4. Paginate: posts[cursor:cursor+limit]
  5. Set next_cursor = last_post.id if more posts exist
